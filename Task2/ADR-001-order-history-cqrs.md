# ADR-001: Просмотр истории заказов через CQRS и материализованную read-модель

## Статус

Accepted

## Контекст

Покупатель NovaMarket должен быстро видеть историю заказов в личном кабинете: список всех заказов, сумму, дату заказа, состав заказа, способ оплаты, способ доставки, финальный статус, ссылки на чеки, возможность оставить отзыв и возможность повторить заказ.

Интерфейс истории заказов должен отвечать быстро, так как это пользовательский сценарий личного кабинета. Пользователь не должен ждать синхронного сбора данных из нескольких сервисов при каждом открытии страницы.

Данные для истории заказов принадлежат разным микросервисам:

- Order Service хранит заказ, состав заказа и жизненный цикл заказа.
- Payment Service хранит данные об оплате и статус платежа.
- Delivery Service хранит способ доставки, трекинг и логистические статусы.
- Catalog Service хранит актуальные и исторически значимые данные о товарах.
- Receipt Service хранит чеки и ссылки на документы.
- Review Service хранит отзывы и признаки доступности отзыва.

Если собирать эти данные синхронно при каждом запросе, интерфейс будет зависеть от доступности и времени ответа нескольких сервисов. Это повышает задержку, усложняет обработку частичных отказов и ухудшает пользовательский опыт.

## Решение

Выбрано решение CQRS + event-driven materialized read model.

Для чтения истории заказов создаётся Order History Query Service. Он не обращается синхронно к Order, Payment, Delivery, Catalog, Receipt и Review сервисам при пользовательском запросе. Вместо этого сервис читает готовую денормализованную read-модель из собственной базы данных.

Read-модель обновляется асинхронно через события Apache Kafka. Основные события для обновления:

- `OrderCreated`
- `PaymentSucceeded`
- `PaymentFailed`
- `DeliveryRequestCreated`
- `OrderDelivered`
- `OrderCancelled`
- `ReceiptGenerated`

Order History Query Service подписывается на эти события, идемпотентно обрабатывает их и обновляет материализованное представление истории заказов. Быстрый пользовательский путь выглядит так:

`Mobile App -> API Gateway -> Order History Query Service -> Order History Read DB`

## Рассмотренные альтернативы

### 1. API Composition

API Composition предполагает, что отдельный сервис или API Gateway при каждом запросе истории заказов синхронно обращается к нескольким микросервисам и собирает итоговый ответ.

Плюсы:

- Простая начальная реализация.
- Нет отдельной read-модели.
- Пользователь получает наиболее актуальные данные из источников.

Минусы:

- Высокая задержка из-за нескольких синхронных вызовов.
- Пользовательский сценарий зависит от доступности Order, Payment, Delivery, Catalog, Receipt и Review сервисов.
- Сложная обработка частичных отказов.
- При росте нагрузки история заказов создаёт дополнительную нагрузку на операционные сервисы.
- Сложнее выполнить требование по быстрому отклику интерфейса.

### 2. CQRS

CQRS разделяет модели записи и чтения. Для истории заказов используется отдельная read-модель, оптимизированная под экран личного кабинета.

Плюсы:

- Быстрые чтения из подготовленной структуры данных.
- Меньше синхронных зависимостей между сервисами.
- Read-модель можно проектировать под конкретные UI-сценарии.
- Операционные сервисы не перегружаются запросами на сложную агрегацию.

Минусы:

- Появляется eventual consistency между write-моделью и read-моделью.
- Нужно поддерживать процесс обновления read-модели.
- Требуется идемпотентная обработка событий и мониторинг отставания проекции.

### 3. Event Sourcing

Event Sourcing предполагает хранение состояния системы как последовательности событий, из которых восстанавливается текущее состояние.

Плюсы:

- Полная история изменений состояния.
- Удобный аудит и возможность восстановления состояния на определённый момент.
- Можно строить разные проекции из единого журнала событий.

Минусы:

- Значительно повышает сложность доменной модели и инфраструктуры.
- Требует строгого управления версиями событий.
- Усложняет разработку, тестирование и поддержку для MVP.
- Не все сервисы NovaMarket требуют полного восстановления состояния из событий.

### 4. CQRS + Event Sourcing

CQRS + Event Sourcing объединяет отдельные модели чтения и записи с хранением источника истины в виде событий.

Плюсы:

- Даёт быстрые read-модели и полную историю изменений.
- Хорошо подходит для сложных доменов с требованиями к аудиту.
- Позволяет перестраивать проекции из событий.

Минусы:

- Самый сложный вариант для реализации и эксплуатации.
- Требует зрелой практики проектирования событий, версионирования и replay-механизмов.
- Увеличивает стоимость разработки MVP.
- Для задачи просмотра истории заказов избыточен на текущем этапе.

## Последствия

Плюсы выбранного решения:

- быстрые чтения;
- меньше синхронных зависимостей между сервисами;
- проще выполнить требование по времени отклика;
- легко подключать аналитику и новые read-модели.

Минусы выбранного решения:

- eventual consistency;
- нужно следить за доставкой событий;
- нужна идемпотентная обработка событий;
- нужна стратегия восстановления read-модели.

Для снижения рисков необходимо:

- хранить идентификаторы обработанных событий или использовать версионирование агрегатов;
- обеспечить повторную обработку событий без дублирования изменений;
- мониторить lag consumer group для Order History Query Service;
- предусмотреть rebuild read-модели из журнала событий или из операционных источников при аварийном восстановлении.

## Итог

Для MVP NovaMarket выбран CQRS с материализованной read-моделью, потому что он обеспечивает быстрый пользовательский отклик и снижает количество синхронных зависимостей между микросервисами. Это решение лучше подходит для экрана истории заказов, чем API Composition, так как не требует собирать данные из нескольких сервисов при каждом запросе пользователя.

Полный Event Sourcing на этапе MVP не выбран, потому что он добавляет существенную инфраструктурную и доменную сложность. Для текущей задачи достаточно событийного обновления read-модели через Kafka без перевода всей системы на хранение состояния исключительно в виде событий.

Выбранный подход оставляет возможность дальнейшего роста: при необходимости можно добавлять новые read-модели, аналитические витрины и дополнительные подписчики на события без изменения пользовательского API истории заказов.
